# Cargo.toml vs Cargo.lock

Cargo uses two complementary files for dependency management. Understanding the difference is essential for reproducible builds.

## Overview

| Aspect | Cargo.toml | Cargo.lock |
|--------|-----------|-----------|
| **Purpose** | Declare dependencies (ranges/constraints) | Record exact versions resolved |
| **Written by** | Developer | Cargo automatically |
| **Edit manually** | Yes | No |
| **Commit to VCS** | Always | Yes for binaries; optional for libraries |

## Cargo.toml

The manifest you author. Declares dependency ranges:

```toml
[package]
name = "hello_world"
version = "0.1.0"

[dependencies]
regex = { git = "https://github.com/rust-lang/regex.git" }
serde = "1.0"
```

## Cargo.lock

Auto-generated by Cargo. Records the exact resolved version of every dependency:

```toml
[[package]]
name = "hello_world"
version = "0.1.0"
dependencies = [
 "regex 1.5.0 (git+https://github.com/rust-lang/regex.git#9f9f693768c584971a4d53bc3c586c33ed3a6831)",
]

[[package]]
name = "regex"
version = "1.5.0"
source = "git+https://github.com/rust-lang/regex.git#9f9f693768c584971a4d53bc3c586c33ed3a6831"
```

## When to Commit Cargo.lock

**Always commit for:**
- Binaries / executables
- Applications (web servers, CLI tools, etc.)

This ensures every developer and CI run uses identical dependency versions.

**Optionally commit for:**
- Libraries published to crates.io
- Library authors often omit it so downstream users aren't constrained

**Reasoning**: When consumers depend on your library, Cargo generates their own `Cargo.lock` with version requirements from your `Cargo.toml`, not your `Cargo.lock`.

## Updating Dependencies

```bash
cargo update            # update all dependencies to latest compatible versions
cargo update regex      # update only the "regex" crate
cargo update --precise regex 1.9.0   # pin to a specific version
```

Running `cargo update` regenerates `Cargo.lock` with newer (but still compatible) versions.

## Reproducible Builds

Without `Cargo.lock`, different developers building the same package at different times may get different dependency versions, leading to inconsistent behavior.

With `Cargo.lock` committed, running `cargo build` always uses the same resolved versions, enabling:
- Reliable `git bisect` for finding regressions
- Consistent CI behavior
- Deterministic production deployments

## Notes

- `cargo install` uses latest dependencies unless you pass `--locked` (uses `Cargo.lock`)
- `cargo build --locked` errors if `Cargo.lock` is out of date (useful in CI)
- `cargo build --frozen` errors if `Cargo.lock` needs updating or network access is required
